# Continue 插件内置工具与二次收紧分析

## 1. 策略判定总览

- 基础策略来源顺序：
  - 用户在 Tools 页面设置的策略（`toolSettings`）
  - 工具定义里的 `defaultToolPolicy`
  - 全局默认值 `allowedWithPermission`
- 二次收紧入口：
  - GUI 侧调用 `tools/evaluatePolicy`
  - Core 侧按工具的 `evaluateToolCallPolicy` 进行动态策略评估
- 核心原则：
  - 动态策略可以更严格，不能更宽松（`disabled > allowedWithPermission > allowedWithoutPermission`）
  - 若基础策略是 `allowedWithPermission`，动态结果即使是 `allowedWithoutPermission` 也不会放宽

关键代码位置：

- `gui/src/redux/thunks/evaluateToolPolicies.ts`
- `core/core.ts`
- `core/tools/policies/fileAccess.ts`
- `packages/terminal-security/src/evaluateTerminalCommandSecurity.ts`

## 2. 二次收紧类型

### 2.1 文件访问类收紧

适用工具会在预处理阶段解析路径并判断是否在工作区内：

- 在工作区内：沿用基础策略
- 工作区外：强制收紧为 `allowedWithPermission`

涉及工具：

- `ls`
- `read_file`
- `read_file_range`
- `create_new_file`
- `view_subdirectory`

相关代码：

- `core/tools/policies/fileAccess.ts`
- `core/util/pathResolver.ts`

### 2.2 终端命令类收紧

`run_terminal_command` 会按命令内容进行安全评估：

- 高危或可疑命令：收紧为 `allowedWithPermission`
- 极高危命令：可能直接 `disabled`
- 安全只读命令：可保持 `allowedWithoutPermission`

相关代码：

- `core/tools/definitions/runTerminalCommand.ts`
- `packages/terminal-security/src/evaluateTerminalCommandSecurity.ts`

### 2.3 前端编辑工具特例

以下 3 个编辑类工具在 GUI 中被直接判定为 `allowedWithoutPermission`：

- `edit_existing_file`
- `single_find_and_replace`
- `multi_edit`

这 3 个工具不走常规的动态收紧分支（属于前端特判）。

相关代码：

- `gui/src/util/toolCallState.ts`
- `gui/src/redux/thunks/evaluateToolPolicies.ts`

## 4. 工具是否出现（可用性）补充

有些内置工具并非总会出现在工具列表中，这与策略不同，是“是否注入配置”的条件问题：

- `search_web`：仅登录后加入
- `view_repo_map` / `view_subdirectory` / `codebase` / `read_file_range`：仅启用实验开关后加入
- `multi_edit` 与 `edit_existing_file + single_find_and_replace`：按模型能力二选一
- `grep_search`：远程工作区不加入

相关代码：

- `core/tools/index.ts`

## 5. 你当前问题的直接结论

当 `ls`、`read_file` 设为 `Automatic` 仍弹授权，最常见根因是：

1. 被判定为“工作区外路径”，触发文件访问二次收紧；
2. 实际调用的工具名与设置项不一致（尤其 MCP 场景）；
3. 打包产物中含旧前端/旧逻辑，导致 UI 显示与运行逻辑不一致。

## 6. Win7 实测优先级建议策略表

说明：

- 目标是“先稳定、再提效”，优先减少误判导致的反复弹窗与风险执行。
- 下表为 Win7 场景建议初始值，可在验证稳定后逐步放宽。
- 术语对照：`Automatic = allowedWithoutPermission`，`Ask First = allowedWithPermission`，`Excluded = disabled`。

### 6.1 P0（高风险，建议先锁死）

| 工具名 (function.name)    | 建议策略                                      | 原因                                     |
| ------------------------- | --------------------------------------------- | ---------------------------------------- |
| `run_terminal_command`    | `Ask First`（生产）/ `Excluded`（高合规场景） | 命令风险最高，且参数变化大               |
| `create_new_file`         | `Ask First`                                   | 写操作，且可能受路径判定影响             |
| `edit_existing_file`      | `Ask First`                                   | 当前前端有 Auto 特判，建议通过治理先收敛 |
| `single_find_and_replace` | `Ask First`                                   | 同上，属于写操作                         |
| `multi_edit`              | `Ask First`                                   | 同上，批量修改风险更高                   |
| `create_rule_block`       | `Excluded`                                    | 默认即禁用，保持即可                     |
| `request_rule`            | `Excluded`                                    | 默认即禁用，保持即可                     |

### 6.2 P1（中风险，建议保守启用）

| 工具名 (function.name)     | 建议策略    | 原因                                       |
| -------------------------- | ----------- | ------------------------------------------ |
| `fetch_url_content`        | `Ask First` | 外网请求，受网络与来源可信度影响           |
| `search_web`               | `Ask First` | 外网检索，结果不可控，建议先人工确认       |
| `read_currently_open_file` | `Ask First` | 涉及当前编辑内容，按最小暴露原则保守处理   |
| `codebase`                 | `Ask First` | 语义检索可能返回较大上下文，先人工确认更稳 |
| `view_repo_map`            | `Ask First` | 工程结构暴露较全，先保守                   |
| `view_subdirectory`        | `Ask First` | 已有路径类二次判定，Win7 下先保守          |

### 6.3 P2（低风险，可优先提效）

| 工具名 (function.name) | 建议策略                | 原因                           |
| ---------------------- | ----------------------- | ------------------------------ |
| `ls`                   | `Automatic`（工作区内） | 只读，且有工作区外自动收紧     |
| `read_file`            | `Automatic`（工作区内） | 只读，且有工作区外自动收紧     |
| `read_file_range`      | `Automatic`（工作区内） | 只读，且有工作区外自动收紧     |
| `file_glob_search`     | `Automatic`             | 只读文件名检索                 |
| `grep_search`          | `Automatic`             | 只读内容检索                   |
| `view_diff`            | `Automatic`             | 只读变更查看                   |
| `read_skill`           | `Automatic`（可选）     | 只读技能文件，可按团队习惯放宽 |

### 6.4 Win7 分阶段落地建议

1. 第一阶段（稳态）：按 P0/P1/P2 全量应用；
2. 第二阶段（观察 3-5 天）：若无异常弹窗与误执行，放宽 `read_currently_open_file`、`codebase`；
3. 第三阶段（提效）：视团队信任度考虑放宽 `search_web`，但仍不建议放宽 `run_terminal_command`。

### 6.5 特别注意（与策略表同等重要）

- 对 `ls/read_file/read_file_range`，即使设为 `Automatic`，工作区外仍会被动态收紧为 `Ask First`，这是预期行为。
- 若出现“明明设了 Auto 仍弹窗”，优先排查：
  - 是否误用绝对路径/URI 导致工作区外判定；
  - 是否命中了非预期工具名（尤其 MCP 工具重命名）；
  - VSIX 是否包含旧的 GUI/Core 产物。

## 7. 一页版（发布说明/运维手册可直接复用）

### 7.1 目标

- 在 Win7 环境优先保证“安全 + 稳定”，再逐步提升自动化比例。
- 默认接受“只读类优先 Auto，写操作与外网/命令类优先 Ask”。

### 7.2 建议策略（推荐初始值）

| 分组               | 工具                                                                                | 建议策略                           |
| ------------------ | ----------------------------------------------------------------------------------- | ---------------------------------- |
| 高风险（P0）       | `run_terminal_command`                                                              | `Ask First`（高合规可 `Excluded`） |
| 高风险（P0）       | `create_new_file` / `edit_existing_file` / `single_find_and_replace` / `multi_edit` | `Ask First`                        |
| 默认禁用（P0）     | `create_rule_block` / `request_rule`                                                | `Excluded`                         |
| 中风险（P1）       | `fetch_url_content` / `search_web`                                                  | `Ask First`                        |
| 中风险（P1）       | `read_currently_open_file` / `codebase` / `view_repo_map` / `view_subdirectory`     | `Ask First`                        |
| 低风险（P2）       | `ls` / `read_file` / `read_file_range`                                              | `Automatic`（仅工作区内）          |
| 低风险（P2）       | `file_glob_search` / `grep_search` / `view_diff`                                    | `Automatic`                        |
| 低风险（P2，可选） | `read_skill`                                                                        | `Automatic`                        |

### 7.3 运维注意事项

- `ls/read_file/read_file_range` 即使是 `Automatic`，在“工作区外路径”仍会被系统动态收紧为 `Ask First`，属正常安全行为。
- 编辑类工具当前有前端 Auto 特判，若你希望严格管控，建议在流程上统一按 `Ask First` 审核执行。
- MCP 场景需按“真实工具名”配置策略（通常为 `serverprefix_tool`），不要只按显示名判断。

### 7.4 快速排障（三步）

1. 先看路径：是否工作区内相对路径；
2. 再看工具名：是否命中了你设置策略的同名工具；
3. 最后看产物：VSIX 是否是最新全量构建（避免旧 GUI/Core 被打包）。
